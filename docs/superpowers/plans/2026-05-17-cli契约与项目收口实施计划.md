# GIS AI TOOL 收口实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 先稳定 CLI 与配置契约，再补齐关键测试覆盖，随后打通 GUI 结果链路并收敛重复入口，使项目进入可持续维护状态。

**Architecture:** 先处理“接口语义不稳定”的底层问题，再补上回归测试，最后收口 GUI 侧的结果展示与入口设计。每一阶段都要求最小化改动范围、明确验证命令、独立提交，避免把多个风险点揉进同一批变更。

**Tech Stack:** C++17, CMake, GoogleTest/CTest, Qt6, SQLite, ONNX Runtime, GDAL/GEOS/PROJ

## 阶段一：修复 CLI 契约与配置错误分类

**目标**

- 统一 `run` / `segment` / `inference` / `batch` 的退出码语义
- 非法 JSON 配置不再落到 `9999`
- 未知 `task_type` 明确失败，不再静默回退到 `segment`

**Files:**
- Modify: `src/cli/main.cpp`
- Modify: `src/fusion/task_config.cpp`
- Modify: `src/fusion/task_config.h`
- Inspect/Optional Modify: `src/core/exception.h`
- Test: `tests/test_cli.cpp`
- Test: `tests/test_fusion.cpp`

- [ ] **Step 1: 在 CLI 测试中新增坏模型退出码断言**

  在 `tests/test_cli.cpp` 中新增三个测试，分别调用：
  - `segment test_data/models/missing_model.onnx test_data/raster/test_100x100.tif`
  - `inference test_data/models/missing_model.onnx test_data/raster/test_100x100.tif`
  - `batch test_data/models/missing_model.onnx test_data/batch_input test_output/batch_cli_probe`

  断言这三个命令的返回码为模型错误码，而不是通用 `1`。

- [ ] **Step 2: 在 CLI 测试中新增坏配置断言**

  在 `tests/test_cli.cpp` 中新增测试，覆盖：
  - `run not_exists.json` -> `4001`
  - `run bad_config.json` -> 配置解析类错误码
  - `run unknown_task_config.json` -> 配置类错误码，并且不是“segment task 缺 model_path”

- [ ] **Step 3: 运行 CLI 测试并确认失败原因正确**

  Run:

  ```powershell
  ctest --test-dir build/release -C Release --output-on-failure -R test_cli
  ```

  Expected:
  - 新增测试失败
  - 失败原因分别指向“返回码仍为 1”“坏 JSON 落到 9999”“未知 task_type 被静默回退”

- [ ] **Step 4: 修改 CLI 子命令返回码**

  在 `src/cli/main.cpp` 中将：

  ```cpp
  return report.success ? 0 : 1;
  ```

  统一改为：

  ```cpp
  return report.success ? 0 : report.error_code;
  ```

  覆盖 `RunSegment`、`RunBatch`、`inference` 分支。

- [ ] **Step 5: 修改配置解析错误分类**

  在 `src/fusion/task_config.cpp` 的 `TaskConfig::LoadFromString` 中捕获 JSON 解析异常，并抛出配置类异常，例如：

  ```cpp
  try {
      nlohmann::json j = nlohmann::json::parse(json_str);
      ...
  } catch (const nlohmann::json::exception& e) {
      throw GisAiConfigException(std::string("Invalid task config JSON: ") + e.what());
  }
  ```

- [ ] **Step 6: 修改未知 task_type 处理逻辑**

  在 `TaskConfig::LoadFromString` 中为任务类型解析增加显式匹配标记：

  ```cpp
  bool matched = false;
  ...
  if (type_str == "segment") {
      config.task_type = TaskType::Segment;
      matched = true;
  }
  ...
  if (!matched) {
      throw GisAiConfigException("Unknown task type: " + type_str);
  }
  ```

- [ ] **Step 7: 运行 CLI 测试验证转绿**

  Run:

  ```powershell
  ctest --test-dir build/release -C Release --output-on-failure -R test_cli
  ```

  Expected:
  - `test_cli` 全绿
  - 所有新增用例通过

- [ ] **Step 8: 做最小手工验证**

  Run:

  ```powershell
  .\build\release\bin\Release\gis_ai_cli.exe run not_exists.json
  .\build\release\bin\Release\gis_ai_cli.exe run test_output\bad_config.json
  .\build\release\bin\Release\gis_ai_cli.exe segment test_data\models\missing_model.onnx test_data\raster\test_100x100.tif
  .\build\release\bin\Release\gis_ai_cli.exe inference test_data\models\missing_model.onnx test_data\raster\test_100x100.tif
  ```

  Expected:
  - 缺配置返回 `4001`
  - 非法 JSON 返回配置类错误码
  - 坏模型返回模型类错误码

- [ ] **Step 9: 提交阶段一**

  ```bash
  git add src/cli/main.cpp src/fusion/task_config.cpp src/fusion/task_config.h tests/test_cli.cpp tests/test_fusion.cpp
  git commit -m "修复CLI契约与配置错误分类"
  ```
