# 单类别二值分割全流程设计

## 1. 背景与目标

当前项目已经具备以下基础能力：

- 基于 ONNX Runtime 的栅格推理能力
- 基于 GDAL 的栅格读写和矢量读写能力
- 简单的整图分割封装 `RasterSeg`
- 面向目录的批处理封装 `BatchProcessor`

但现有流程仍然停留在“小图整图缩放后推理”的阶段，和当前目标存在明显差距：

- 不支持大图分块推理
- 输出结果仍以 `shp` 为主，不符合更稳妥的交付方式
- 缺少单图级结果汇总与批量级汇总
- 批处理能力复用了旧的整图推理路径，无法支撑大图稳定闭环

本阶段目标只做一件事：构建一个稳定、可重复、可批量执行的单类别二值分割主链路，直接支持：

- 大图推理
- 二值栅格输出
- 矢量化输出
- 单张输入
- 目录批量输入

本阶段不追求平台化和高性能极限，优先保证主链路正确、职责清晰、结果可追溯。

## 2. 范围约束

### 2.1 本阶段要做

- 支持大图按块推理
- 支持整图合并后的二值栅格输出
- 支持整图结果统一矢量化输出
- 支持单张影像处理
- 支持目录批量处理
- 输出单图级汇总信息
- 输出批量级汇总信息

### 2.2 本阶段不做

- 多类别分割
- 实例分割
- 复杂形态学后处理
- 多模型自动适配
- 分块后分别矢量化再拼接
- GUI
- 训练能力
- 分布式或多机推理

## 3. 方案比较

### 方案 A：在现有 `RasterSeg` 上直接堆逻辑

做法：

- 继续沿用 `RasterSeg`
- 在其中加入切块、读取、合并、导出、矢量化
- `BatchProcessor` 继续调用它

优点：

- 初期改动文件少
- 短时间内能跑起来

缺点：

- `RasterSeg` 会迅速变成多职责类
- 单图流程与批量流程耦合更深
- 后续扩目录批量、摘要输出、预览输出会越来越乱

结论：

不推荐。短期看似快，后期会拖慢整个项目。

### 方案 B：保留现有能力，补一条新的大图主链路

做法：

- 保留 `RasterSeg` 作为旧接口兼容层
- 新增大图分块推理相关模块
- 批处理改为调用新的单图主流程

优点：

- 风险可控
- 便于逐步替换旧逻辑
- 单图与批量可共享同一条主链路
- 后续扩展 Python、CLI、服务接口更顺

缺点：

- 初期文件数会增加
- 需要先把边界划分清楚

结论：

推荐。最符合当前“先做稳整条流程”的目标。

### 方案 C：直接重写整个融合层

做法：

- 废弃 `RasterSeg` 与 `BatchProcessor`
- 重新定义分割流程、结果结构和外部接口

优点：

- 结构最整洁
- 可以一步到位按新边界建设

缺点：

- 风险最高
- 回归成本高
- 当前项目基础还不够稳，全面重写收益不高

结论：

不推荐。当前阶段没有必要。

## 4. 选型结论

采用方案 B。

核心原则如下：

- 新流程优先服务“单类别二值分割”
- 单图和批量共享同一条核心主链路
- 先整图合并，再统一矢量化
- 先把结果做对、做稳、做完整，再考虑进一步优化

## 5. 对外接口设计

第一版同时保留两层接口：

- 以 CLI 作为主入口
- 以 C++ 库接口作为内部稳定能力层

### 5.1 CLI 入口

统一命令形态：

```bash
gis_dl_tool segment --input <path> --output <dir> --model <model.onnx>
```

建议支持的核心参数：

- `--input`
- `--output`
- `--model`
- `--tile-size`
- `--overlap`
- `--threshold`
- `--batch-size`
- `--polygon-min-area`
- `--compress`

约定：

- `input` 为文件时执行单张处理
- `input` 为目录时执行目录批量处理

### 5.2 C++ 内部接口

建议形成稳定的能力入口：

```cpp
SegmentationResult RunSegmentation(const SegmentationOptions& options);
BatchSegmentationResult RunBatchSegmentation(const SegmentationOptions& options);
```

原则：

- 批量模式复用单图模式
- 不为批量单独发明完全不同的参数结构

## 6. 核心参数设计

第一版只保留对结果和可用性最关键的参数：

- `input`
- `output`
- `model`
- `tile_size`，默认 `1024`
- `overlap`，默认 `128`
- `threshold`，默认 `0.5`
- `batch_size`，默认 `1`
- `vectorize`，默认开启
- `polygon_min_area`，默认 `0`
- `compress`，默认 `LZW`

说明：

- `stride` 不直接暴露，由内部通过 `tile_size - overlap` 计算
- 第一版不增加更多预处理和后处理参数
- 真正影响结果与体验的核心参数只有 `tile_size`、`overlap`、`threshold`

## 7. 模块边界设计

### 7.1 `TilingPlanner`

职责：

- 根据整图宽高、`tile_size`、`overlap` 生成切块窗口
- 处理边界块

不负责：

- 实际读取影像
- 模型推理
- 栅格写出

### 7.2 `TileReader`

职责：

- 按切块窗口从原始栅格读取数据
- 生成单块 `RasterData`
- 保留块级地理变换信息

不负责：

- 切块规划
- 推理
- 拼接

### 7.3 `TileInferenceRunner`

职责：

- 持有模型会话
- 对单块执行推理
- 输出块级概率图或分割结果

不负责：

- 切块读取
- 结果拼接
- 导出文件

### 7.4 `TileMerger`

职责：

- 将多个块结果合并为整图结果
- 处理重叠区融合
- 输出完整尺寸结果

第一版融合策略：

- 若输出概率图，则重叠区简单平均
- 合并完成后统一阈值化为二值掩膜

### 7.5 `SegmentationExporter`

职责：

- 写出整图二值栅格 `mask.tif`
- 基于整图结果矢量化输出 `polygon.gpkg`
- 输出 `summary.json`

第一版不要求必须输出预览图，若实现成本低可作为附加项补上，但不作为主目标。

### 7.6 `BatchSegmentationRunner`

职责：

- 枚举目录中的输入影像
- 针对每张影像调用单图主流程
- 汇总结果并输出 `batch_summary.csv`

原则：

- 目录批量只是单图流程外层的一层循环
- 第一版默认串行执行即可

### 7.7 `SegmentationPipeline`

职责：

- 串联单图完整处理过程
- 统一单图级错误处理
- 生成单图级运行结果

调用顺序：

```text
TilingPlanner
  -> TileReader
  -> TileInferenceRunner
  -> TileMerger
  -> SegmentationExporter
```

## 8. 数据结构设计

建议至少引入以下结构：

### 8.1 `SegmentationOptions`

对外总配置，包含：

- 输入路径
- 输出路径
- 模型路径
- `tile_size`
- `overlap`
- `threshold`
- `batch_size`
- `vectorize`
- `polygon_min_area`
- `compress`

### 8.2 `TileWindow`

用于描述单个切块：

- `x`
- `y`
- `width`
- `height`

### 8.3 `SegmentationResult`

用于描述单图处理结果：

- 是否成功
- 是否部分成功
- 输入路径
- 输出栅格路径
- 输出矢量路径
- 错误信息
- tile 数量
- 耗时

### 8.4 `BatchSegmentationResult`

用于描述批量处理结果：

- 总文件数
- 成功数
- 部分成功数
- 失败数
- 单图结果列表

## 9. 输入输出结构

### 9.1 单张模式

```text
output/
  image_a_mask.tif
  image_a_polygon.gpkg
  image_a_summary.json
```

说明：

- `mask.tif` 为主输出，必须保留空间参考和仿射变换
- `polygon.gpkg` 为整图掩膜统一矢量化后的输出
- `summary.json` 记录输入、参数、耗时、状态和输出路径

### 9.2 批量模式

```text
output/
  image_a/
    image_a_mask.tif
    image_a_polygon.gpkg
    image_a_summary.json
  image_b/
    image_b_mask.tif
    image_b_polygon.gpkg
    image_b_summary.json
  batch_summary.csv
```

说明：

- 每张图单独占一个子目录
- `batch_summary.csv` 用于记录每张图的状态、耗时、错误原因和结果路径

## 10. 异常处理策略

### 10.1 输入阶段异常

包括：

- 输入路径不存在
- 输入目录为空
- 影像无法打开
- 影像没有有效波段
- 影像缺少必要元信息

策略：

- 单张模式直接失败并返回明确信息
- 批量模式记录失败项并继续处理下一张

### 10.2 模型阶段异常

包括：

- 模型文件不存在
- 模型加载失败
- 输入输出张量不匹配
- 推理会话初始化失败

策略：

- 视为全局阻塞错误
- 直接终止当前任务

### 10.3 切块与拼接异常

包括：

- 切块结果为空
- 某块推理失败
- 合并尺寸不一致

策略：

- 单张模式直接失败
- 批量模式当前图失败，其他图继续

### 10.4 输出阶段异常

包括：

- 栅格写出失败
- 矢量化失败
- 汇总文件写出失败

策略：

- `mask.tif` 优先级最高
- 若栅格成功、矢量失败，标记为部分成功
- 批量汇总需区分 `success`、`partial_success`、`failed`

## 11. 测试策略

本阶段测试重点是流程正确性，不是模型精度评估。

### 11.1 单元测试

重点覆盖：

- `TilingPlanner`
- overlap 与步长计算
- 边缘块生成
- 输出命名规则
- 批量输入枚举

### 11.2 集成测试

重点覆盖：

- 单张影像完整流程
- 输出二值栅格尺寸正确
- 空间参考保留正确
- 矢量文件可正常打开

### 11.3 批处理测试

重点覆盖：

- 目录内多张影像处理
- 单张失败不拖垮全批次
- `batch_summary.csv` 字段完整

### 11.4 回归测试

重点覆盖：

- 同一输入同一参数的输出尺寸稳定
- 汇总结构稳定
- 主链路未被后续修改破坏

## 12. 极简实施方案

第一批改动只做以下内容：

1. 在 `fusion` 层补齐大图分块主链路的数据结构与模块骨架
2. 让 `RasterIO` 支持按窗口读取和按指定尺寸写出整图结果
3. 新增整图拼接逻辑，先采用简单平均融合
4. 新增统一导出逻辑，输出 `mask.tif`、`polygon.gpkg`、`summary.json`
5. 将批处理改为调用新的单图主流程
6. 补最关键的单元测试和流程测试

第一批不做的事情：

- 不改训练相关内容
- 不加 GUI
- 不做复杂后处理
- 不碰性能优化和并行调度细节
- 不优先处理当前编译依赖阻塞

## 14. 结论

当前项目最需要的不是继续堆功能点，而是尽快形成一条真实可用的主链路。

单类别二值分割是最合适的切入点，因为它足够聚焦，能以最低复杂度把以下几件最重要的事情一次性打通：

- 大图处理骨架
- 栅格输出规范
- 矢量化输出规范
- 批处理容错机制
- 结果追溯能力

等这条链路稳定之后，再扩多类别、后处理、服务化或界面层，成本会明显更低。
