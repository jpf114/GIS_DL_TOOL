# GIS AI 算法库下一阶段开发优先级清单

## 一、文档目的

本文档用于回答：

- 未来 1 个月应该优先做什么
- 未来 3 个月应形成哪些阶段成果
- 未来 6 个月应朝什么方向扩展

本清单基于以下原则排序：

1. 先解决实用性瓶颈，再扩功能面
2. 先形成完整工作流，再扩零散算子
3. 先确保可复现和可交付，再谈平台化

## 二、优先级总原则

当前阶段的优先级判断如下：

### 第一优先级：工程闭环

如果项目不能稳定构建、测试和安装，那么新增功能的价值会被明显削弱。

### 第二优先级：核心场景打透

如果没有一个真正能用的完整场景，项目很难建立实用价值。

### 第三优先级：生态与交付增强

当核心场景成立后，再增强格式支持、配置系统、CLI 和服务化入口，投入产出比更高。

## 三、未来 1 个月优先级

### P0：解决构建与验证闭环

目标：

- 本地可配置
- 测试链路可执行
- 安装链路可验证

建议事项：

1. 解决 `vcpkg` baseline 和本机依赖环境问题
2. 完成 `cmake --preset=dev-windows` 的成功配置
3. 编译 `generate_test_data` 工具并验证夹具生成
4. 验证 `test_io_integration`
5. 验证 `test_ai_integration`
6. 验证安装后外部示例可配置

原因：

这部分是后续一切功能建设的基础。如果现在跳过，后续每一项开发都会被环境问题反复打断。

### P1：补齐测试资产闭环

目标：

- 栅格、矢量、模型测试资产路径明确
- 测试脚本行为统一

建议事项：

1. 明确 `test_seg_model.onnx` 的标准来源
2. 如果可能，补仓库内自动生成方式
3. 如果暂时不能自动生成，至少固定模型来源标准与导入流程

原因：

当前 AI 测试仍缺最后一块闭环，这会持续影响验证效率。

### P2：完成安装与消费链路验证

目标：

- `cmake --install` 可运行
- `find_package(gis_ai CONFIG REQUIRED)` 可工作

建议事项：

1. 验证安装导出目标
2. 验证 `examples/installed_package`
3. 根据结果修正包导出或文档

原因：

这是项目从“源码库”变成“可被其他工程使用的 SDK”的关键节点。

## 四、未来 3 个月优先级

### P0：实现遥感大图分割完整流程

这是我认为未来 3 个月最重要的事情。

目标：

- 从“小图/测试图推理”升级到“真实遥感大图处理”

建议补齐能力：

1. 滑窗切片推理
2. overlap 融合
3. 边缘平滑
4. 结果拼接恢复
5. NoData 处理
6. 可配置 tile size / stride
7. 可配置后处理阈值

原因：

一旦这条能力形成，项目就会从“算法库”明显升级为“可工作的遥感解译引擎”。

### P1：加强结果后处理与质量控制

目标：

- 输出结果更接近真实业务使用要求

建议补：

1. 小斑块过滤
2. 面积阈值过滤
3. 多边形简化
4. 拓扑修复
5. 属性回写
6. 结果质量摘要

原因：

真实业务里，原始掩膜输出往往不能直接交付，后处理质量会直接影响用户体验。

### P2：增强批处理能力

目标：

- 从“单任务运行”升级到“任务批量执行”

建议补：

1. 输入目录扫描
2. 批量任务配置
3. 失败重试
4. 运行摘要
5. 输出目录规范

原因：

如果没有批处理，用户无法真正把它纳入日常生产流程。

### P3：补配置化运行入口

建议补：

1. JSON/YAML 任务配置
2. 配置校验
3. 参数模板
4. 默认配置示例

原因：

这能显著降低项目的使用门槛，也方便未来 CLI 和服务化。

## 五、未来 6 个月优先级

### 方向 1：增强数据生态兼容

建议优先补：

1. GeoPackage
2. COG
3. LAS/LAZ 标准支持
4. 多波段影像映射策略

### 方向 2：增加标准使用入口

建议补：

1. CLI 工具
2. 统一任务配置驱动
3. 错误码手册
4. 日志与运行报告

### 方向 3：扩展模型与任务类型

建议补：

1. 多模型管理
2. 变化检测
3. 分类任务
4. 更多后处理模板

### 方向 4：评估服务化与平台化

只有在前面几项稳定之后，才建议推进：

1. 本地服务接口
2. 任务队列
3. 插件式任务注册
4. 更完整的外部集成能力

## 六、不建议当前优先投入的方向

### 1. 不建议优先做很多零散算法

理由：

- 增加维护成本
- 提升测试复杂度
- 对用户实用价值提升有限

### 2. 不建议过早做复杂平台化

理由：

- 当前核心流程尚未打透
- 平台化会带来较大架构和运维负担
- 容易掩盖核心价值还未建立的问题

### 3. 不建议同时铺多个业务方向

理由：

- 容易分散资源
- 产出不聚焦
- 难形成可展示、可交付的强能力

## 七、专业建议版排序

如果以“投入产出比”排序，我会建议按下面顺序推进：

### 立即执行

1. 解决本地构建与 `vcpkg` 阻塞
2. 完成测试夹具与测试模型准备闭环
3. 完成安装与外部消费验证

### 第一阶段

1. 大图滑窗推理
2. 结果拼接
3. 后处理规则
4. 批量处理增强

### 第二阶段

1. 配置驱动任务
2. CLI 入口
3. GeoPackage / COG 支持
4. 运行报告与错误体系

### 第三阶段

1. 更多模型任务
2. 服务化入口
3. 插件化任务扩展

## 八、阶段验收建议

### 1 个月验收标准

- 可成功配置和编译开发环境
- 测试夹具路径清晰
- 关键测试可运行
- 安装链路至少完成一次本地验证

### 3 个月验收标准

- 可完成大图分割到结果输出的完整流程
- 可批量处理输入数据
- 输出结果具备基本后处理能力

### 6 个月验收标准

- 有标准 CLI 或任务入口
- 可支持更多现实 GIS 数据格式
- 可作为团队内部通用引擎进行复用

## 九、结论

当前最重要的不是“继续堆更多功能”，而是：

1. 把工程链路补实
2. 把遥感解译核心场景打透
3. 把使用入口做出来

如果只能选一件最有价值的事，我会选：

**优先把“遥感大图语义分割全流程”做成真正可跑、可批量、可输出、可交付的能力。**

因为一旦这一条链路成立，项目的实用价值会出现质变。
