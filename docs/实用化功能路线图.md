# GIS AI 算法库实用化功能路线图

## 一、文档目的

本文档用于回答三个核心问题：

1. 当前项目处于什么阶段
2. 从实用化角度，项目未来应重点补强哪些能力
3. 应该沿着什么产品与技术路线演进，才能从“算法库雏形”走向“可落地的遥感智能解译引擎”

## 二、当前项目定位

从现有代码结构、公共 API、测试组织和文档成熟度来看，当前项目已经不是纯概念验证，而是一套具备较完整技术骨架的 GIS + AI 基础库。  
它已经覆盖了：

- Core 基础能力
- Raster / Vector / PointCloud IO
- 常见 GIS 算法
- ONNX Runtime 推理封装
- 遥感栅格分割融合流程
- C API 与 Python ctypes 验证入口

但如果从“用户今天能不能拿它做实际生产任务”来判断，项目目前仍偏向：

- 技术验证版
- 开发者底层库
- 可演示，但未形成高频生产工作流

因此，本项目当前最准确的定位不是“生产级平台”，而是：

**面向遥感智能解译方向的 GIS AI 基础引擎雏形**

## 三、核心判断

当前项目给人的“功能不多”感受，并不是因为完全没有能力，而是因为现有功能主要集中在底层算子与模块能力，缺少以下三类更容易被用户直接感知的内容：

### 1. 缺少完整业务流程

目前已有较多原子功能，例如：

- 矢量缓冲、裁剪、相交、简化
- 栅格重采样、归一化、阈值分割
- ONNX 模型加载与栅格推理
- 掩膜转面

但真实用户通常不关心单个算子，而关心完整流程，例如：

- 大幅遥感影像切片推理与拼接
- 批量解译与结果汇总
- 面结果清洗与拓扑修复
- 输出结果入库和统一导出

目前项目在“算法能力”层面已经具备基础，但在“工作流能力”层面仍偏弱。

### 2. 缺少现实 GIS 生态兼容性

当前数据支持仍偏基础，离生产环境常见生态还有差距。  
典型差距包括：

- 缺少 GeoPackage 等现代矢量格式优先支持
- 缺少 COG 等面向云和大规模影像场景的栅格支持策略
- 点云能力仍较基础
- ONNX 模型接入路径还不够工程化

### 3. 缺少交付与使用层能力

即便底层算法可用，如果没有以下能力，项目仍难以在团队中被真正使用：

- 配置文件驱动任务
- 批处理任务入口
- 命令行工具
- 安装后外部消费路径
- 完整错误码与运行报告
- 稳定的测试资产与验证流程

## 四、推荐发展主线

如果从未来方向上做选择，我不建议把项目继续做成“算法杂货铺”，而应该尽快收敛成一条清晰主线。

### 推荐主线：遥感智能解译引擎

建议把项目定位为：

**以遥感影像智能解译为核心场景的轻量级 GIS AI 引擎**

这条路线最适合当前项目，原因有三点：

1. 当前已有的 AI + 栅格 + 矢量化能力天然适合遥感场景
2. 遥感解译具备明确的业务流程，可形成高频使用链路
3. 在保持库化设计的同时，也能逐步演进到 CLI、服务化和平台化

### 不建议的路线

短期内不建议把项目扩成“大而全 GIS 平台”或“全场景 AI 算法市场”。  
那样会导致范围失控、功能发散、测试复杂度飙升，而且很难形成真正有竞争力的核心能力。

## 五、实用化能力建设方向

### 方向 1：补完整的遥感分割生产流程

这是当前最值得优先投入的方向。

建议补齐：

- 大图滑窗推理
- 重叠区域融合
- NoData 跳过策略
- 结果拼接恢复
- 面积阈值过滤
- 孤立小斑块去除
- 多边形简化与拓扑修复
- 输出统计摘要

一旦这条链路打通，项目就不再只是“能跑单张图的小库”，而是开始具备真实解译引擎的价值。

### 方向 2：补批量任务能力

业务场景里最常见的不是“处理 1 个文件”，而是：

- 处理一个目录
- 处理一个项目批次
- 对失败项重试
- 导出统一报告

因此建议补齐：

- 批量任务扫描
- 输出目录规范
- 失败重试
- 进度追踪
- 批量日志
- 批处理报告

### 方向 3：补更现实的数据格式与生态支持

建议按价值优先级补：

- GeoPackage
- COG
- 更规范的 LAS/LAZ 路径
- 多波段影像选择与映射
- 多模型输入输出配置

### 方向 4：补配置化与交付能力

建议新增：

- JSON/YAML 任务配置
- 配置校验
- 统一参数模板
- CLI 入口
- 错误码说明
- 日志与运行报告

## 六、阶段性演进建议

### 阶段 A：工程闭环阶段

目标：

- 构建稳定
- 测试可复现
- 安装可验证
- 文档可信

这是项目从“能开发”迈向“能持续迭代”的前提。

### 阶段 B：单场景打透阶段

目标：

- 打透遥感大图语义分割全流程
- 建立一条可以被业务验证的完整能力链

这是项目从“技术库”迈向“业务引擎”的关键阶段。

### 阶段 C：产品化能力阶段

目标：

- 增加 CLI
- 增加服务化接口
- 增加标准配置任务入口
- 增加安装与部署说明

这个阶段开始让项目具备更强的组织内推广能力。

## 七、专业建议

如果我是该项目的技术负责人，我会给出以下建议：

### 建议 1：不要急于扩很多算法

当前真正的短板不是“算法数量太少”，而是“可落地工作流太少”。  
继续加一批零散算子，带来的实用价值有限，反而会分散精力。

### 建议 2：优先做一个能打的核心场景

我会明确把“遥感大图语义分割全流程”作为第一核心场景，围绕它补齐所有缺失能力。  
这比同时铺多个方向更容易建立成果。

### 建议 3：把文档、测试、安装视为产品能力

对这种底层引擎项目来说：

- 可复现构建
- 可复现测试
- 清晰文档
- 清晰错误信息

并不是附属工作，而是产品能力本身。

### 建议 4：保持库化架构，但逐步增加使用入口

底层保持库化没有问题，但应该逐步补：

- CLI
- 标准配置任务入口
- 安装后示例消费工程

这样才能让项目既适合研发集成，也适合独立使用。

## 八、路线图结论

一句话总结：

**当前项目不应再优先追求“更多零散功能”，而应优先补齐“遥感解译完整流程能力 + 可交付工程能力”。**

未来的推荐路线是：

1. 先完成工程闭环
2. 再打透遥感大图解译流程
3. 再补 CLI、配置化和外部消费能力
4. 最后再扩更多模型、格式和服务化能力

这样项目的发展路径会更稳，也更容易真正进入实际使用场景。
